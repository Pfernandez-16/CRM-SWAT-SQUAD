SEGMENTATION DECISION (Claude's Discretion):
- servicio_interes in fact_leads does NOT contain Manufacturers/Individuals — it has values like 'Consultoria'
- tipo_membresia in fact_calificacion DOES contain 'Manufacturers', 'Individuals', 'Attraction'
- Use tipo_membresia from fact_calificacion as the segmentation source
- Leads without a calificacion row or with null/empty tipo_membresia go into "Total" only (not Manufacturers or Individuals)
- "Attraction" leads should also be excluded from Manufacturers/Individuals (they are a third category that goes into Total only)

INCONTACTABLES MAPPING:
- The requirement says "Duplicado, Equivocado, SPAM" but cat_opciones Status Lead has: 'Duplicado', 'Invalido'
- Map: Duplicado -> 'Duplicado', Equivocado -> 'Invalido' (closest match), SPAM -> no status exists
- For SPAM: count leads where calidad_contacto = 'Malo' as a proxy, OR simply report 0 with a note
- Best approach: Use the 3 labels from the requirement (Duplicado, Equivocado, SPAM) as display names, map Duplicado to status='Duplicado', Equivocado to status='Invalido', SPAM to a count of 0 (no data source exists yet). This is honest and the frontend can display it.

  <action>
Create gas-crm-project/Analytics.js with all the foundation infrastructure and the Embudo General section. The file structure should be:

**1. File header comment** explaining this is the analytics module, deployed alongside Codigo.js via clasp.

**2. Helper functions (private, suffix with _):**

- `filterByDateRange_(leads, dateIn, dateOut)` — Filter leads array where fecha_ingreso is between dateIn and dateOut (inclusive). Parse dates with `new Date(ts).getTime()`. Handle NaN dates by excluding them. Return filtered array.

- `buildLeadInteractionsIdx_(interacciones)` — Build a one-to-many index: `{ id_lead: [interaction1, interaction2, ...] }`. Return the map object. This is different from indexBy_ which is 1:1.

- `getSegment_(lead, calificacionIdx)` — Given a lead and the calificacion index, return segment string. Logic: look up `calificacionIdx[String(lead.id_lead)]`, read its `tipo_membresia`. Return `'manufacturers'` if tipo_membresia is 'Manufacturers', `'individuals'` if 'Individuals', `'other'` for 'Attraction' or anything else, `'unknown'` if no calificacion or empty tipo_membresia. Always lowercase the return value.

- `calcPercentage_(part, total)` — Return `(part / total * 100).toFixed(1)` as a string, or `'0.0'` if total is 0.

- `calcDelta_(currentVal, previousVal)` — Return `((currentVal - previousVal) / previousVal * 100).toFixed(1)` as string. If previousVal is 0 and currentVal > 0, return `'100.0'`. If both are 0, return `'0.0'`.

- `buildSegmentedMetric_(leads, calificacionIdx, metricFn)` — Apply metricFn to all leads (total), then to leads where getSegment_ returns 'manufacturers', then 'individuals'. Return object:
  ```javascript
  {
    total: { count: N, pct: '100.0' },
    manufacturers: { count: M, pct: calcPercentage_(M, N) },
    individuals: { count: I, pct: calcPercentage_(I, N) }
  }
  ```

- `buildSegmentedMetricWithDelta_(currentLeads, previousLeads, calificacionIdx, metricFn)` — Same as buildSegmentedMetric_ but also runs metricFn on previousLeads for each segment, then adds `delta` field using calcDelta_ to each segment. Return:
  ```javascript
  {
    total: { count: N, pct: '100.0', delta: deltaStr },
    manufacturers: { count: M, pct: pctStr, delta: deltaStr },
    individuals: { count: I, pct: pctStr, delta: deltaStr }
  }
  ```

- `buildSegmentedAmountWithDelta_(currentLeads, previousLeads, calificacionIdx, amountFn)` — Same pattern but amountFn returns a numeric sum (not a count). For amounts, `pct` is not meaningful so omit it. Return `{ total: { amount: X, delta: D }, manufacturers: { amount: Y, delta: D }, individuals: { amount: Z, delta: D } }`.

**3. Section calculators:**

- `calculateEmbudoGeneral_(currentLeads, previousLeads, allLeads, contactosIdx, interaccionesIdx, calificacionIdx, deals)` — Implements all 13 locked funnel metrics from CONTEXT.md. Each metric is a function passed to buildSegmentedMetricWithDelta_. Return an object with keys matching metric names:

  Metric functions (each takes a leads subset and returns a count):
  - **totalLeads**: `leads.length`
  - **contactables**: Count leads where contactosIdx lookup has non-empty telefono_1 OR email (trim + check non-empty string)
  - **contactados**: Count leads that have at least 1 entry in interaccionesIdx
  - **conRespuesta**: Count leads that have at least 1 interaction with resultado === 'Contesto' (use exact accent: 'Contesto' — verify the actual string from the data, which is stored as Unicode)
  - **dialogoCompleto**: Count leads that have 2+ consecutive 'Contesto' toques. Logic: get all interactions for lead, filter to resultado='Contesto', sort by numero_toque ascending, check if any adjacent pair has toque numbers differing by exactly 1. A lead qualifies if it has at least one such consecutive pair.
  - **dialogoIntermitente**: Count leads that have at least 1 'Contesto' interaction but do NOT qualify as dialogoCompleto. This is mutually exclusive with dialogoCompleto.
  - **interes**: Count leads where calificacionIdx lookup has mostro_interes_genuino === 'Si' (check exact accent from data)
  - **descartados**: Count leads with status === 'Perdido'
  - **asignadosVentas**: Count leads with status === 'Paso a Ventas'
  - **carryOver**: Special metric — from `allLeads` (not date-filtered), find leads where fecha_ingreso < dateIn AND status === 'Paso a Ventas' AND fecha_asignacion is between dateIn and dateOut. Use allLeads parameter, not the filtered subset.
  - **montosInversion**: Sum of monto_proyeccion from deals linked to period leads (via id_lead match). Use buildSegmentedAmountWithDelta_.
  - **dealsCerrados**: Count of deals linked to period leads where status_venta === 'Vendido'. Use buildSegmentedMetricWithDelta_.
  - **montoCierres**: Sum of monto_cierre from deals linked to period leads where status_venta === 'Vendido'. Use buildSegmentedAmountWithDelta_.

  For carryOver, the delta comparison uses the SAME logic but shifted: previous-carry-over = leads with fecha_ingreso < prevDateIn AND status='Paso a Ventas' AND fecha_asignacion between prevDateIn and prevDateOut.

  IMPORTANT: For deal-based metrics (montosInversion, dealsCerrados, montoCierres), build a `leadIdSet` of current period lead IDs, then filter deals where `deal.id_lead` is in that set. For segmentation, look up the LEAD's segment (not the deal's).

  Return format:
  ```javascript
  {
    totalLeads: { total: {...}, manufacturers: {...}, individuals: {...} },
    contactables: { total: {...}, manufacturers: {...}, individuals: {...} },
    // ... all 13 metrics
  }
  ```

- `calculateIncontactables_(currentLeads, previousLeads, calificacionIdx)` — Three sub-metrics using buildSegmentedMetricWithDelta_:
  - **duplicado**: leads with status === 'Duplicado'
  - **equivocado**: leads with status === 'Invalido' (mapped from "Equivocado" in the requirement)
  - **spam**: count 0 always (no data source exists — return `{ total: { count: 0, pct: '0.0', delta: '0.0' }, manufacturers: { count: 0, pct: '0.0', delta: '0.0' }, individuals: { count: 0, pct: '0.0', delta: '0.0' } }`)

Since readTable_ preserves the original value, compare with `=== 'Contesto'` (with tilde on o if the sheet has it). To be safe, also log the actual resultado values encountered during development.
